# GeminiClaw / AI Team OS 통합 기획안

## 1. 문서 목적
이 문서는 `geminiclaw` 프로젝트를 **회사 운영용 AI Team OS**로 발전시키기 위한 제품 기획 기준서입니다.
하나의 AI API Key(GPT 또는 Gemini 중 택1)를 연결한 뒤, 회사의 각 부서를 역할별 AI 팀으로 구성하고, 사용자는 전용 대시보드 또는 외부 채널을 통해 업무를 지시·승인·추적할 수 있는 시스템을 설계 대상으로 합니다.

---

## 2. 제품 정의

### 2.1 한 줄 정의
회사의 각 부서를 역할별 AI 팀으로 구성하고, 사용자는 채팅형 명령창과 실시간 대시보드를 통해 업무를 지시하며, AI 팀은 내부적으로 기획·분석·개발·검토·보고를 수행하고, 사람은 승인 및 오프라인 실행만 담당하는 **회사 운영용 AI 오케스트레이션 플랫폼**.

### 2.2 핵심 전제
- 시작 시점에는 **AI Provider 1개 + API Key 1개**만 연결한다.
- 여러 부서형 AI 팀은 서로 다른 모델이 아니라, **같은 모델 엔진에 역할·권한·SOP·도구·메모리 범위를 다르게 부여한 팀 단위 실행 구조**로 구현한다.
- 채널 입력, 대시보드 입력, 승인, 로그, 결과 보고를 모두 **Task 중심 단일 흐름**으로 통합한다.

### 2.3 제품 철학
- AI는 업무를 수행하고, 사람은 책임과 승인을 담당한다.
- 모든 업무는 추적 가능해야 한다.
- 외부에서 보낸 메시지도 내부에서는 정규화된 작업(Task)으로 관리한다.
- 위험 작업은 반드시 승인 단계를 거친다.
- 결과물은 일회성 대화가 아니라 지식 자산으로 축적한다.

## 3. 해결하려는 문제

### 3.1 현재 회사 운영의 문제
- 대표 또는 핵심 담당자가 너무 많은 업무를 직접 해석하고 분배해야 한다.
- 지시, 진행상황, 승인, 결과물이 여러 메신저와 문서 도구에 흩어진다.
- 외부에 나가 있을 때 업무 진행 상태를 한눈에 보기 어렵다.
- 반복적인 기획, 정리, 보고, 검토, 로그 확인 업무에 시간이 많이 든다.
- 과거 의사결정과 중간 산출물의 추적이 어렵다.

### 3.2 AI Team OS가 제공하는 해결 방식
- 자연어 지시를 **Task**로 정규화한다.
- Task를 AI 부서 조직에 자동 배분한다.
- 진행 흐름을 노드 그래프와 상태 카드로 시각화한다.
- 승인 요청을 대시보드 및 채널로 동기화한다.
- 모든 중간 판단과 결과를 로그 및 지식 문서로 저장한다.

## 4. 제품 목표

### 4.1 비즈니스 목표
- 대표 1인의 수동 조율 시간을 대폭 줄인다.
- 업무의 누락과 중복을 줄인다.
- 외부 이동 중에도 회사 운영이 지속되도록 한다.
- 회사 지식과 업무 흐름을 시스템 자산으로 축적한다.

### 4.2 운영 목표
- 업무 지시 후 5초 이내에 Task가 생성된다.
- 승인 필요 작업은 즉시 알림으로 전환된다.
- 모든 작업은 프로젝트 단위로 로그와 결과를 남긴다.
- 외부 채널 메시지는 1회 수신 후 내부 Task와 연결된다.

### 4.3 사용자 경험 목표
- 사용자는 복잡한 메뉴 대신 **채팅창 한 곳**에서 지시할 수 있어야 한다.
- 진행 흐름은 그래픽으로 쉽게 파악되어야 한다.
- 승인 여부는 팝업 또는 채널에서 즉시 처리 가능해야 한다.
- 과거 작업은 일지처럼 쉽게 검색·열람 가능해야 한다.

## 6. 주요 사용 시나리오

### 6.1 대시보드 직접 지시
1. 대표가 메인 채팅창에 업무를 입력한다.
2. 시스템이 업무를 이해하고 Task로 등록한다.
3. PM Agent가 세부 단계로 분해한다.
4. 관련 AI 부서가 순차 또는 병렬 수행한다.
5. 진행 상황이 그래프 및 상태 카드에 반영된다.
6. 승인 필요 시 팝업이 뜬다.
7. 승인 후 결과가 보고서로 정리된다.
8. 로그와 지식 문서에 저장된다.

### 6.2 외부 이동 중 채널 지시
1. 사용자가 Telegram/WhatsApp/Slack에서 지시를 보낸다.
2. Channel Hub가 메시지를 수신한다.
3. 메시지는 내부 Task로 변환된다.
4. Control Agent가 업무를 분해한다.
5. 중간 결과 및 최종 결과가 해당 채널로 회신된다.
6. 승인 요청이 발생하면 채널에서 승인/반려한다.

### 6.3 현장 사진/문서 업로드
1. 외부 담당자가 채널 또는 대시보드로 사진/PDF를 업로드한다.
2. 시스템이 파일을 프로젝트 지식 저장소에 연결한다.
3. Operations 또는 Knowledge Agent가 내용을 분석한다.
4. 요약 및 후속 작업이 자동 생성된다.
5. 필요한 경우 추가 확인 요청이 채널로 발송된다.

## 7. 핵심 기능

### 7.1 Command Chat
- 자연어 지시 입력
- AI 팀 호출 결과 요약
- 실행 전 작업 요약 표시
- 연속 대화 기반 추가 지시 지원

### 7.2 Workflow Live Graph
- 노드형 작업 흐름 시각화
- 현재 진행 중 노드 강조
- 승인 대기 노드 표시
- 노드 클릭 시 상세 로그 열람

### 7.3 Approval Center
- 고위험 작업 승인 요청 목록
- 팝업 승인
- 채널 승인
- 반려 사유 기록
- 승인 이력 보존

### 7.4 Work Journal / Logs
- 작업별 일지 기록
- 전체 이벤트 타임라인
- 입력, 중간 산출물, 최종 결과 보존
- 검색 및 필터 지원

### 7.5 Knowledge Library
- 프로젝트별 문서 저장
- PDF/이미지/메모 업로드
- AI 자동 요약 및 태그 부여
- 관련 Task와 연결

### 7.6 Channel Hub
- Telegram / WhatsApp / Slack / Email 수신
- 승인 요청 전송
- 외부 메시지 → 내부 Task 변환
- 첨부파일 수신 후 후속 처리

### 7.7 Department Manager
- 부서 AI 활성화/비활성화
- 역할 설명 관리
- SOP 및 프롬프트 관리
- 승인 권한 범위 관리

## 8. 운영 원칙

### 8.1 승인 원칙
다음 작업은 승인 필수:
- 외부 발송
- 운영 정책 변경
- 실서비스 배포
- 고객에게 확정 답변 발송
- 계약/정산/법무 관련 결정
- 서버 파괴적 명령

### 8.2 로그 원칙
모든 Task는 아래 내용을 남긴다.
- 지시 원문
- 해석 결과
- 분배된 부서
- 중간 산출물
- 승인 요청 및 결과
- 오류와 재시도 이력
- 최종 보고서

### 8.3 권한 원칙
- AI는 실행 가능 범위 내에서만 동작한다.
- 승인 없는 파괴적 작업은 금지한다.
- 채널 사용자는 역할 기반 제한을 받는다.

### 8.4 지식 축적 원칙
- 결과물은 프로젝트/부서/카테고리 기준으로 정리한다.
- 단순 채팅 결과도 가치 있으면 문서로 승격한다.
- 과거 유사 작업은 재사용 가능해야 한다.

## 10. 시스템 아키텍처 개요

### 10.1 핵심 구성
- Frontend: Flutter Desktop / Web
- Backend: 오케스트레이터 서버(FastAPI 또는 Node)
- Realtime: WebSocket
- Storage: PostgreSQL + 파일 스토리지 + 검색 인덱스
- AI Provider Adapter: GPT 또는 Gemini 단일 연결
- Channel Connector: Telegram/WhatsApp/Slack/Email

### 10.2 오케스트레이터 역할
- Task 생성 및 상태 관리
- 부서별 AI 실행 순서 조정
- 승인 단계 삽입
- 로그 기록
- 채널 송수신 중계
- 실시간 이벤트 브로드캐스트

### 10.3 단일 API Key 운용 방식
- 시스템 설정에서 Provider 1개 선택
- API Key 1개 저장
- 모든 부서 Agent는 동일 Provider를 사용
- 부서별 차이는 모델 자체가 아니라 **역할/프롬프트/도구/지식 범위**로 만든다.

## 14. 최종 정리
이 시스템은 단순한 AI 챗봇이 아니라, **회사의 각 부서를 역할별 AI 팀으로 구성하여, 업무 지시·실행·승인·기록·채널 연동까지 하나의 흐름으로 통합하는 Company AI Control Tower**입니다.

시작은 API Key 1개로 가능하지만, 제품의 본질은 모델 수가 아니라 **오케스트레이션 구조, 승인 체계, 로그, 지식 저장소, 채널 통합**에 있습니다.
