# Mozzy II 개발 계획서

  

- 작성일: 2026-03-26

- 기준 문서: `Mozzy(모지) 백서 Ver 4.2 - Global Expansion Edition 작성일_ 2026년 3월 26일.md`

- 기준 버전 정의: 현재 완성된 Mozzy 앱은 `Mozzy Version 1`로 명명한다.

- 작업 브랜치: `Mozzy_II` 생성 완료

  

## 1. 문서 목적

  

본 문서는 현재 완성된 `Mozzy Version 1`을 기준선으로 삼아, 차세대 제품인 `Mozzy II`를 별도 제품 라인으로 재정의하고 실제 개발 착수가 가능한 수준의 제품 방향, 구조, 팀/에이전트 구성, 브랜치 운영, 폴더 구조, UI/UX 방향, 단계별 실행 계획을 정리하기 위한 개발 계획서이다.

  

`Mozzy I`가 하이퍼로컬 슈퍼앱을 29개국 25개 언어를 지원하는 `Global One Build`로 완성한 버전이라면, `Mozzy II`는 반대로 `하이퍼로컬에서 출발하지만 로컬 포스트가 글로벌로 자연스럽게 확장되고, 이웃 연결이 국가 경계를 넘어서 강화되는 앱`을 목표로 한다.

  

## 2. Mozzy I 기준선 정리

  

`Mozzy I`는 다음 상태를 이미 달성한 완성 버전이다.

  

- 11개 핵심 feature 기반 하이퍼로컬 슈퍼앱 완성

- 전 feature 공통 지도 기반 탐색 경험 제공

- 29개국, 25개 언어 지원을 전제로 한 `Global One Build` 확정

- 로컬 뉴스, 구인구직, 분실물, 마켓플레이스, 가게, 모임, 친구 찾기, 부동산, 경매, POM, Together를 하나의 앱에서 운영

- 다국어 및 번역, 국가 단위 검색, 로컬/국가/글로벌 탐색 구조를 이미 구현

  

Mozzy I의 강점은 `하나의 코드베이스로 여러 국가를 서비스하는 실행력`이다. 다만 Mozzy II는 여기서 한 단계 더 나아가 `로컬 포스트의 글로벌 확장성`, `국가 간 이웃 연결성`, `통합된 추천·번역·신뢰 레이어`, `더 선명한 제품 아이덴티티`를 강화해야 한다.

  

## 3. Mozzy II 제품 정의

  

### 3.1 한 줄 정의

  

Mozzy II는 `Local-first, Global-ready Neighborhood Network`를 지향하는 하이퍼로컬-글로벌 연결형 슈퍼앱이다.

  

### 3.2 핵심 컨셉

  

- 사용자는 동네 기준으로 글을 올린다.

- 시스템은 그 글의 성격을 분석해 `동네`, `도시`, `국가`, `글로벌`까지 확장 가능한 공개 범위를 관리한다.

- 로컬 포스트는 자동 번역, 지도 기반 재노출, 국가 간 추천, 관심사 기반 연결을 통해 글로벌 이웃에게 도달한다.

- Mozzy II는 `지역 게시판의 글로벌 버전`이 아니라 `지역성에서 출발한 글로벌 생활 네트워크`를 만든다.

  

### 3.3 Mozzy II 북극성

  

`어느 나라 어느 동네에서 올린 생활 포스트든, 필요한 사람에게는 국경을 넘어 발견되고 연결된다.`

  

## 4. Mozzy I 대비 Mozzy II의 핵심 진화

  

### 4.1 제품 관점

  

- Mozzy I: 11개 기능을 하나의 앱 안에서 제공하는 글로벌 하이퍼로컬 슈퍼앱

- Mozzy II: 11개 기능을 유지하되 `지역 포스트의 글로벌 순환 구조`를 제품의 중심 엔진으로 재설계

  

### 4.2 구조 관점

  

- Mozzy I: feature 중심 통합

- Mozzy II: `App Shell + Discovery Layer + Geo Layer + Trust Layer + Feature Domains` 중심 모듈 구조

  

### 4.3 사용자 경험 관점

  

- Mozzy I: 사용자가 위치를 바꾸며 feature를 탐색

- Mozzy II: 사용자가 위치를 중심으로 시작하되, 시스템이 국가·언어·관심사·이동 계획을 고려해 `다음 연결`을 제안

  

### 4.4 데이터 관점

  

- Mozzy I: 로컬 포스트 저장 및 지역 기반 검색

- Mozzy II: `GeoScope`, `Reach`, `Translation`, `Trust`, `Signal`, `Global Relay`를 포함한 확장형 포스트 모델로 재설계

  

## 5. Mozzy II 목표

  

### 5.1 제품 목표

  

- 로컬 포스트의 글로벌 발견 가능성을 높인다.

- 29개국 25개 언어 구조를 유지하되 국가 간 연결 경험을 강화한다.

- 11개 feature를 `공통 탐색 체계` 아래 재정렬한다.

- 현재 repo에서 Mozzy I와 충돌 없이 병행 개발 가능한 구조를 만든다.

- Mozzy II 전용 폴더 구조와 디자인 시스템을 선행 구축한다.

  

### 5.2 비즈니스 목표

  

- 국가별 신규 유저가 로컬 콘텐츠 부족으로 이탈하는 문제를 줄인다.

- 로컬 콘텐츠를 국가 단위와 글로벌 단위로 확장하여 네트워크 효과를 만든다.

- 이주민, 유학생, 여행자, 교민, 글로벌 노마드에게 더 강한 실사용 이유를 제공한다.

- 커머스, 구인구직, 커뮤니티, 이벤트의 교차 전환을 높인다.

  

## 6. Mozzy II 핵심 제품 원칙

  

### 6.1 Local-first, Global-readable

  

모든 포스트는 로컬 맥락에서 작성되지만, 기본적으로 글로벌 탐색과 자동 번역을 고려한 데이터 구조를 가져야 한다.

  

### 6.2 Post once, distribute progressively

  

사용자는 한 번 작성한다. 시스템은 포스트 성격과 설정에 따라 로컬, 도시, 국가, 글로벌 노출을 단계적으로 확장한다.

  

### 6.3 Geography is a first-class entity

  

지역은 단순 필터가 아니라 탐색, 추천, 신뢰, 번역, 알림, 광고, 랭킹을 결정하는 핵심 엔티티다.

  

### 6.4 Shared contract across all features

  

모든 feature는 동일한 `LocationFilterContract`, `MapCardContract`, `SearchContract`, `EmptyStateContract`, `TranslationContract`를 사용해야 한다.

  

### 6.5 Trust before scale

  

글로벌 확장은 신뢰 레이어가 동반되지 않으면 성립하지 않는다. 프로필 신뢰도, 인증, 신고, AI 검수, 거래 보호, 운영 툴이 같이 설계되어야 한다.

## 8. Mozzy II 핵심 시나리오

  

### 8.1 로컬 포스트의 글로벌 확장

  

- 발리에서 올린 Local News가 국가 단위 추천으로 확장된다.

- 그중 글로벌 수요가 높은 글은 번역된 글로벌 요약 카드로 재노출된다.

- 해외 유저는 관심 국가, 이주 예정 지역, 여행 예정지, 언어 관심사에 따라 해당 포스트를 탐색한다.

  

### 8.2 이웃의 글로벌 연결

  

- Find Friends, Together, Clubs는 `근처 사람 찾기`를 넘어 `같은 언어권`, `같은 관심사`, `같은 이동 경로`, `같은 국가 관심사`를 가진 사람을 연결한다.

- 한 지역의 생활 정보와 사람이 다른 국가의 준비 단계 유저에게 연결된다.

  

### 8.3 feature 간 교차 순환

  

- Local News에서 본 행사 글이 Together로 이어진다.

- Jobs에서 본 채용 글이 Nearby Shops, Marketplace, Real Estate 소비 흐름으로 이어진다.

- POM은 지역 자랑에서 끝나지 않고 `여행`, `이주`, `상권`, `커뮤니티`의 입구가 된다.

  

## 9. UI/UX 방향

  

Mozzy II는 최신 모바일 UX 흐름을 반영하되, 과도한 장식보다 `명확한 계층`, `적응형 레이아웃`, `지도와 피드의 일관된 전환`, `국가/언어/지역 상태의 명료한 표현`에 집중한다.

  

### 9.1 UI/UX 원칙

  

- 앱 전역에 명확한 시각 계층을 만든다.

- 지도, 리스트, 상세, 작성, 프로필의 전환 구조를 통일한다.

- 작은 화면에서는 빠른 1차 행동, 큰 화면에서는 정보 밀도와 병렬 탐색을 강화한다.

- edge-to-edge 레이아웃과 safe area 대응을 기본으로 한다.

- 검색, 필터, 언어, 위치, 국가 상태는 항상 상단에서 이해 가능해야 한다.

- 주요 액션은 문맥 가까이에 배치하고, 탭/세그먼트/필터 칩은 반복 패턴으로 통일한다.

- 44pt 이상 터치 타깃, 충분한 대비, 다국어 줄바꿈, 길이 변화, RTL 가능성까지 고려한다.

- 예측 가능한 back navigation과 자연스러운 motion을 적용한다.

  

### 9.2 Mozzy II 비주얼 방향

  

- 기본 톤: clean, editorial, geo-aware, trust-first

- 지도/피드 결합형 레이아웃을 브랜드 시그니처로 발전

- 카드 디자인은 `지역`, `언어`, `신뢰`, `확장 범위`를 빠르게 읽을 수 있게 설계

- 아이콘보다 정보 구조가 먼저 드러나는 타이포 중심 설계

- 색상은 feature 색을 유지하되, 글로벌/국가/로컬 상태는 공통 semantic token으로 통합

  

### 9.3 권장 IA

  

- Home

- Explore

- Map

- Create

- Inbox

- My

  

각 feature는 이 1차 shell 아래 들어가며, 개별 feature 진입 후에도 검색/지도/필터/생성 흐름은 공통 규칙을 따른다.

## 12. 데이터 모델 재설계 방향

  

Mozzy II 포스트는 기존 `locationParts + countryCode` 구조를 유지하되 다음 필드를 강화한다.

  

- `geoScope`: local, city, national, global

- `reachMode`: local_only, national_expandable, global_expandable

- `translationState`: none, machine_ready, reviewed

- `sourceLanguage`

- `targetLanguages`

- `trustScore`

- `verificationFlags`

- `signalScore`

- `relayTargets`

- `mapVisibility`

- `discoveryChannels`

  

이 구조를 통해 모든 포스트는 `어디서 작성되었는가`뿐 아니라 `어디까지 노출될 수 있는가`, `누가 신뢰할 수 있는가`, `어떤 언어로 소비 가능한가`를 명시적으로 갖게 된다.

  

## 13. 개발 운영 전략

  

### 13.1 브랜치 전략

  

- 메인 기준선: `main`

- Mozzy II 메인 개발 브랜치: `Mozzy_II`

- 작업 브랜치 규칙:

  - `Mozzy_II/design-system`

  - `Mozzy_II/app-shell`

  - `Mozzy_II/discovery`

  - `Mozzy_II/geo-search`

  - `Mozzy_II/trust`

  - `Mozzy_II/domain-local-news`

  - `Mozzy_II/domain-marketplace`

  

### 13.2 병행 개발 원칙

  

- Mozzy I와 Mozzy II는 초기에는 공존한다.

- Mozzy II는 현재 프로덕션 코드를 직접 대체하지 않고 독립 구조로 축적한다.

- 공통 모듈은 검증 후 역으로 Mozzy I에 일부 재도입 가능하도록 설계한다.

- Shell, Design System, Shared Contract가 안정되기 전까지 feature 마이그레이션을 서두르지 않는다.

  

## 14. 에이전트 구성 제안

  

Mozzy II는 한 명의 메인 구현자 아래 다수의 전문 에이전트 또는 담당 워크스트림이 병렬로 움직이는 방식이 적합하다.

  

### 14.1 총괄 에이전트

  

#### A1. Product Architect Agent

  

- 책임: 전체 제품 구조, 단계별 우선순위, 도메인 간 경계 설정

- 산출물: 제품 구조도, 도메인 분리 기준, 마일스톤 기준

  

#### A2. Technical Lead Agent

  

- 책임: 아키텍처, 공통 계약, 코드 규칙, 품질 게이트

- 산출물: 폴더 구조, 상태관리 기준, 데이터 계약, merge 기준

  

### 14.2 UI/UX 에이전트

## 19. 글로벌 유저 유지 전략

  

아래 전략은 `Mozzy I 백서`, Apple의 구독 및 앱 비즈니스 모델 가이드, Google Play Billing 문서, AdMob native ads 문서, Stripe Connect의 marketplace/platform monetization 구조를 참고해 Mozzy II에 맞게 재구성한 제안안이다.

  

### 19.1 공통 유지 엔진

  

- `Geo Follow Loop`: 사용자가 국가, 도시, 동네를 팔로우하고 해당 지역의 새 포스트, 이벤트, 상점, 채용, 매물 업데이트를 주기적으로 받도록 한다.

- `Language Follow Loop`: 사용자는 자신의 언어권 외에도 관심 언어권 피드를 선택할 수 있어야 하며, 번역된 요약과 원문 전환을 자연스럽게 제공해야 한다.

- `Travel / Relocation Loop`: 여행 예정지, 이주 예정지, 유학 예정지, 취업 희망 국가를 저장하면 해당 지역 feature가 자동으로 개인화된다.

- `Trust Loop`: 인증, 후기, 거래 성공, 이벤트 참석, 채용 응답, 그룹 운영 이력을 축적해 국가가 달라도 신뢰가 이어지게 만든다.

- `Creator / Pro Loop`: 자주 올리는 유저, 상인, 채용 담당자, 호스트, 중개인, 크리에이터에게 전용 툴과 통계를 제공해 반복 활동을 유도한다.

- `Cold Start Relief`: 특정 지역에 포스트가 적으면 전국 검색, 인접 도시, 같은 언어권, 글로벌 relay 콘텐츠를 자동 제안해 빈 피드 이탈을 줄인다.

## 20. feature별 수익화 전략 및 원칙

  

### 20.1 수익화 원칙

  

- `디지털 혜택`은 Apple/Google 정책에 맞춰 subscription, consumable, non-consumable 중심으로 설계한다.

- `실물/현장/서비스 거래`는 국가별 결제 사업자 및 marketplace 결제 구조를 통해 거래 수수료와 payout 비즈니스를 만든다.

- `광고`는 무작위 배너보다 native ad, sponsored card, promoted placement 중심으로 설계한다.

- `신뢰와 도구`는 반복 사용자가 비용을 지불하는 B2B/B2Pro 모델로 발전시킨다.

- `무료 핵심 사용성`은 유지하고, 속도·노출·관리·분석·인증 같은 부가가치를 유료화한다.

  

### 20.2 feature별 수익화 설계

  

| Feature | 1차 수익화 | 2차 수익화 | 실행 메모 |

| :-- | :-- | :-- | :-- |

| Local News | 지역 기관/상점용 promoted notice, sponsored local card | verified local authority 페이지, 국가별 digest sponsorship | 사용자 작성 기본 노출은 무료 유지, 공공/상업 정보만 명확히 분리 유료화 |

| Lost & Found | urgent boost, recovery reward escrow 운영 수수료 | 여행 허브 제휴 스폰서십, verified recovery badge | 공익 기능 성격이 강하므로 과금은 속도·검증·제휴 중심으로 제한 |

| Clubs | organizer subscription, promoted club/event placement | 멤버십 관리 도구, 협찬 패키지, 정기 회비 관리 수수료 | 활발한 운영자는 관리도구와 노출 패키지에 지불 가능성 높음 |

| Find Friends | premium verification pass, advanced filter pass | 여행자 networking pack, profile highlight boost | 과금은 관계 자체보다 안전성과 탐색 편의성에 두어야 함 |

| Real Estate | agent/broker subscription, featured listing, qualified lead fee | relocation package, neighborhood analytics, premium contact unlock | 국가별 부동산 광고 시장과 이주 수요를 동시에 겨냥 가능 |

| Auction | listing fee, final value fee, featured lot placement | reserve price tools, seller verification, buyer protection fee | 신뢰와 희소성 기반 고마진 구조 설계 가능 |

### 20.3 Mozzy II 통합 매출 포트폴리오

  

- `Subscription Revenue`: recruiter, merchant, seller pro, organizer, broker, creator 도구 구독

- `Transaction Revenue`: marketplace, auction, event ticketing, lead, payout, instant payout 수수료

- `Advertising Revenue`: native ads, sponsored cards, geo-targeted promoted placement

- `Trust Revenue`: verification pass, premium identity, seller/recruiter/host badge

- `Growth Revenue`: boosts, featured slots, urgent exposure, regional spotlight pack

  

### 20.4 우선순위 제안

  

초기 Mozzy II는 모든 feature를 동시에 유료화하지 않는다. 다음 순서가 현실적이다.

  

1. `Marketplace + Auction + Jobs + Nearby Shops`에서 거래/노출/B2B 과금을 먼저 만든다.

2. `Together + Real Estate + Clubs`에서 host/agent/organizer용 recurring tool 매출을 붙인다.

3. `POM + Local News`는 광고와 sponsored placement를 붙여 discovery monetization으로 확장한다.

4. `Find Friends + Lost & Found`는 신뢰/안전/편의성 중심의 제한적 유료화를 적용한다.

  

## 21. 결론

  

Mozzy I는 `29개국 25개 언어를 지원하는 하이퍼로컬 슈퍼앱의 완성본`이다. Mozzy II는 이를 단순 개선하는 버전이 아니라, `로컬 포스트의 글로벌 확장`과 `국경을 넘는 이웃 연결`을 제품의 중심 엔진으로 재정의하는 차세대 버전이어야 한다.

  

따라서 Mozzy II는 다음 순서를 반드시 지켜야 한다.

  

1. Mozzy II 전용 shell과 design system을 먼저 만든다.

2. 공통 위치/검색/지도/빈 상태 계약을 선행 확립한다.

3. 핵심 feature 몇 개를 먼저 옮겨 제품 정체성을 검증한다.

4. 이후 11개 feature 전체로 확장한다.

  

이 계획에 따라 `Mozzy_II` 브랜치에서 폴더 구조, UI/UX, 공통 계약, 핵심 feature 시범 구현까지 단계적으로 진행할 수 있다.
