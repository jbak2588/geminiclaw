I. 서비스 개요 (Service Overview)
1.1 프로젝트 소개 (Project Introduction)
Mozzy(모지)는 인도네시아, 한국, 미국, 유럽 등 전 세계 29개 주요 국가를 하나의 앱(One App)으로 연결하는 Global One Build 기반의 하이퍼로컬 커뮤니티 슈퍼앱입니다.
기존의 파편화된 로컬 서비스들과 달리, 하나의 앱(One App)으로 전 세계 어느 도시에서든 해당 지역에 최적화된 ▲중고거래 ▲동네 커뮤니티 ▲구인구직 ▲부동산 ▲소상공인 서비스를 끊김 없이 이용할 수 있습니다.
Mozzy은 "Global Standard Technology, Local Specialized Experience"를 지향하며, 기술적으로는 단일화하되 경험적으로는 철저히 현지화된 서비스를 제공합니다.
1.2 서비스 비전과 미션 (Vision & Mission)
●      Vision: 언어와 국경의 장벽 없이 지역 사회를 신뢰 있게 연결하는 글로벌 커뮤니티 플랫폼.
●      Mission:
○      Global Connectivity: 단일 앱으로 29개국을 연결하는 경계 없는 로컬 경험 제공.
○      No Language Barrier: AI 실시간 번역 기술을 통해 이주민, 여행객, 현지인 간의 소통 장벽 제거.
○      Trust & Safety: AI 검수 기술로 중고거래 및 정보의 신뢰성 확보.
○      Visual Exploration: 지도 기반의 직관적인 데이터 탐색 경험 제공.

II. Global One Build 핵심 전략 (Key Strategy)
Mozzy는 하나의 소스 코드(Single Codebase)로 다국가 서비스를 동시에 운영하며, 특히 지도 기반의 데이터 시각화 기술을 통해 국경 없는 정보 탐색을 지원합니다.
2.1 언어 장벽 없는 소통 (AI-Powered Translation)
●      On-Device AI Translation: Google ML Kit를 내장하여 서버 비용 없이 게시글 및 채팅을 실시간 번역합니다.
●      Seamless UX: 한국어 ↔ (인도네시아어/베트남어/영어) 간 즉시 변환으로 언어 장벽을 제거합니다.
2.2 전 세계 데이터 시각화 (Unified Global Map Experience) [NEW]
●      Shared Map Browser 기술: 구인구직, 부동산, 중고거래, 동네소식 등 앱 내 11개 모든 핵심 기능에 공통된 지도 브라우징 기술(SharedMapBrowser)을 적용했습니다.
●      Scalable Exploration: 단순 리스트 뷰의 한계를 넘어, 동(Neighborhood) 단위 $\leftrightarrow$ 시(City) 단위 $\leftrightarrow$ 국가(National) $\leftrightarrow$ 전 세계(Global) 단위로 지도를 줌인/줌아웃하며 모든 데이터를 직관적으로 확인할 수 있습니다.
●      Border-less UX: 사용자는 현재 위치뿐만 아니라, 이주 예정인 국가나 여행지의 ▲일자리 ▲매물 ▲축제 정보를 지도 이동만으로 미리 탐색할 수 있어 글로벌 확장에 최적화되어 있습니다.
2.3 국가별 자동 최적화 (Dynamic Localization)
●      Auto-Routing System: 접속 국가(IP/GPS)에 따라 locations_id, locations_kr 등 국가별 DB로 자동 라우팅되어 속도와 법적 리스크를 관리합니다.
2.4 통합 결제 시스템 (Multi-PG Integration)
●      One Payment Gateway: Xendit(동남아), PortOne(한국) 등 국가별 결제 시스템을 하나의 매니저로 통합 관리합니다.

VI. 글로벌 확장: 29개국 행정구역 DB & Dual Track System
6.1 Dual Track Location System (이원화 위치 데이터 구조)
Mozzy는 인도네시아 기존 구조를 보존하면서도, 다른 국가 확장 비용을 최소화하기 위해 Dual Track 데이터 구조를 채택합니다.
• Track 1 (Legacy Optimized - Indonesia): Provinsi → Kabupaten → Kecamatan → Kelurahan 4단계 구조를 유지하며 약 83,500개 수준의 방대한 마을 데이터를 보존
• Track 2 (Global Standard V2 - 28개국): L1(주/광역) → L2(도시/구) → L3(동네/우편번호/워드 등)로 매핑 가능한 유연한 표준 계층 구조 적용
6.2 Country Only Query & Firestore Sharding
• locations_id, locations_kr 등 국가별 컬렉션(또는 서브컬렉션) 구조로 샤딩하여 성능과 운영 리스크를 분리
• 사용자가 국가를 선택하면 해당 국가의 데이터만 필터링하여 보여주는 Country Only 쿼리 최적화 적용

I. Local News / Berita Lokal / 뉴스·공지
1. 기획의도
1.1 필요성 / Purpose
지역 뉴스와 공공 공지, 커뮤니티 소식 등을 한데 모아 지역 주민 간 정보 교류를 원활하게 하는 공간입니다. 공공성과 신뢰성을 바탕으로, 누구나 뉴스형 게시글을 등록할 수 있으며, 기관/기업/개인 모두를 위한 공지 플랫폼 역할을 합니다. 기존 게시판 형태를 벗어나 태그 기반의 스마트한 분류, 명확한 위치/시간 정보가 담긴 이벤트 카드 형태로 진화하여 사용자 경험을 향상시킵니다.
1.2 주요 구성 / Key Components
●      게시 대상:
○      지방 정부, 공공 기관: 검증 후 무료 게시 (공공성 확보)
○      일반 사용자: 결혼, 출산, 개업 등 개인 소식 등록 가능
○      기업/광고주: 유료 공지 기능 제공 (예: 상단 고정, 강조 효과)
●      게시물 유형:
○      지방 정부 공지사항 및 정책 안내
○      공공 기관 캠페인 및 행사 정보
○      인도네시아 뉴스 포털 지역 뉴스 (자동 수집 고려)
○      일반 주민 소식 (축하, 부고, 행사, 나눔 등)
●      기능 구조:
○      태그 기반 분류: #공사, #축제, #교통 등 키워드 중심의 직관적인 분류 제공 (자동 추천 기능 포함)
○      이벤트 특화: 행사 일시 및 장소 정보를 구조화하여 캘린더 연동 및 지도 표시 지원
○      고정 공지: 관리자 또는 유료 사용자를 위한 상단 고정 기능
○      신고/스팸 필터링: 쾌적한 커뮤니티 환경 조성을 위한 관리 시스템
1.3 핵심 시스템 설명 / System Logic
●      권한 기반 등록: 마을 대표/기관 관리자 계정은 별도 인증을 통해 무제한 무료 등록 권한 부여.
●      스마트 태그 시스템: 게시글 작성 시 본문 내용을 분석하여 관련 태그를 자동으로 추천해 주는 TagRecommender 도입.
●      위치 기반 노출: 사용자 프로필(UserModel)의 위치 정보(locationName, geoPoint)를 기반으로 "우리 동네" 소식을 우선적으로 노출 및 필터링.
●      이벤트 정보 구조화: 단순 텍스트가 아닌 eventDate, eventAddress, geoPoint 필드를 통해 행사 정보를 명확하게 저장하고 활용.
●      게시물 관리: 사용자 신고 누적 시 자동 숨김 처리 및 관리자 검토 대기 전환.
1.4 기술 아키텍처 / Technical Architecture
●      데이터 구조 (Firestore):
○      Collection: posts
○      주요 필드:
■      category: AppCategories 상수로 관리 (뉴스, 행사, 질문 등)
■      tags: 검색 및 필터링을 위한 태그 리스트
■      eventDate: 행사 날짜 및 시간 (DateTime)
■      eventAddress: 행사 장소명 (String)
■      geoPoint: 지도상 정확한 좌표 (GeoPoint)
■      locationName: 게시글 작성 위치 또는 타겟 지역 명칭
■      (삭제됨: searchIndex, eventTime, location String)
●      기능 구현:

[개정 완료] 3. Lost & Found / 분실물 찾기 (V2.0 Final)
1. 개요 (Overview)
●      목적: 지역 사회(Kelurahan/Kecamatan) 내에서 발생하는 분실물과 습득물을 신속하게 연결하고, 이웃 간의 도움을 **'신뢰 점수(Trust Score)'**와 **'따뜻한 후기(Testimonial)'**로 시각화하여 커뮤니티의 선순환 구조를 구축함.
●      핵심 가치: 단순 정보 공유를 넘어, 실제 물건을 찾아준 이웃에게 **보상(사례금)**과 **명예(미담)**를 동시에 제공.

2. 주요 기능 및 UX 프로세스 (Key Features & User Flow)
2.1. 리스트 & 카드 (List & Card UI)
●      시각적 구분 (Color Coding):
○      LOST (분실): 긴급함을 강조하는 Red 계열 테두리 및 배지.
○      FOUND (습득): 안도감을 주는 Green/Blue 계열 테두리 및 배지.
●      상태 오버레이 (Status Overlay):
○      해결 완료 시 이미지 위에 직관적인 오버레이 텍스트 표시.
○      Lost 해결 시: "찾았어요! (FOUND IT!)"
○      Found 해결 시: "돌려줬어요! (RETURNED!)"
●      빈 화면 처리 (Empty State):
○      선택된 지역에 게시물이 없을 경우, 사용자 이탈 방지를 위해 "전체 지역 보기(필터 초기화)" 버튼 제공.
2.2. 등록 및 수정 (Create & Edit)
●      사례금(Bounty) 강화:
○      Rp 10,000 단위 콤마(,) 자동 포맷팅 적용.
○      유효성 검사: 최소 금액(10k) 및 최대 금액 제한을 통해 장난성 게시물 방지.
●      위치 설정: 현재 위치 자동 인식 및 지도 핀 설정을 통한 정확한 좌표(GeoPoint) 저장.
2.3. 상세 화면 & 소통 (Detail & Interaction)
●      1:1 채팅 연동: 게시물 하단 버튼을 통해 즉시 채팅방 개설 (자동으로 물건 정보가 컨텍스트로 전달됨).
●      미담(후기) 섹션:
○      해결 완료된 게시물에는 **"따뜻한 후기 카드"**가 상단에 노출됨.
○      후기 내용: 도움 받은 사연(예: "비 오는데 직접 가져다 주셨어요!").
○      프로필 연결: 도움을 준 이웃의 닉네임이 파란색 링크로 표시되며, 클릭 시 해당 유저의 프로필로 이동.
2.4. 해결 프로세스 (Resolution Workflow) - Core Feature
작성자가 '해결 완료' 버튼을 누를 때의 흐름:
대상 선택: 이 게시물과 채팅 기록이 있는 이웃 목록이 자동으로 뜸.
옵션 선택:
○      특정 이웃 선택: 앱 내 채팅을 통해 도움을 준 사람 지정.
○      앱 외부/오프라인 해결: "앱 외부 이웃과 해결했어요" 선택 (대상 없음).
후기 작성: 감사의 마음을 담은 짧은 메시지 입력 (선택 사항).
보상 및 피드백 (System Action):
○      신뢰 점수 지급: 선택된 이웃에게 Trust Score +10, Thanks Count +1 자동 지급 (트랜잭션 처리).
○      시스템 메시지: 해당 채팅방에 "이웃이 도움을 확정했습니다. 신뢰 점수가 지급되었습니다!" 자동 전송.

4.7 중고거래 (Marketplace / Pre-Loved Items)
1 핵심 목표: AI를 통한 신뢰 혁신
Mozzy(모지)의 중고거래 기능은 단순한 C2C 연결을 넘어, AI 기술을 통해 인도네시아 중고 시장의 고질적인 '신뢰 문제'를 해결하는 것을 핵심 목표로 합니다. 이를 위해 **V2.1 (동적 AI 검수)**과 **V2.2 (AI 현장 인수)**라는 두 가지 핵심 시스템을 도입합니다.
2 AI 검수 (V2.1): 판매자의 신뢰 확보
판매자는 상품 등록 시 AI 검수를 선택하여 구매자에게 객관적인 'AI 검증 보고서'를 제공할 수 있습니다.
1차 분석 (증거 제안): 판매자가 상품 사진을 등록하면(product_registration_screen.dart), 백엔드(initialproductanalysis)는 ai_rules_v2.json의 카테고리별 '추천샷(suggestedShots)' DB를 기반으로, AI에게 "부족한 증거(missing_evidence_list)"를 식별하도록 요청합니다.
동적 증거 보강: ai_evidence_suggestion_screen.dart는 이 '부족한 증거 목록'(예: "신발 밑창 샷", "스마트폰 IMEI 화면")을 사용자에게 제시하며, 사진을 추가하거나 건너뛸 수 있는 선택권을 제공합니다.
최종 보고서 생성: generatefinalreport 함수는 모든 사진과 '건너뛴 항목' 목록을 취합하여, notes_for_buyer (구매자 참고사항)가 포함된 최종 보고서를 생성하고, 이를 product_model.dart의 aiReport 필드에 저장합니다.
비교 검증 (개편안 2): product_detail_screen.dart는 구매자에게 **1) 판매자가 직접 작성한 원본 설명(description)**과 **2) AI가 분석한 객관적인 보고서(AiReportViewer)**를 의도적으로 동시에 노출합니다. 이를 통해 구매자는 두 정보를 비교하며 신뢰도를 판단할 수 있습니다.
3 AI 인수 (V2.2): 구매자의 안전 확보
AI 검증이 완료된 상품은 'AI 안심 예약' 및 '현장 검증' 대상이 됩니다.
10% 예약금 결제 (1단계): 구매자는 product_detail_screen.dart의 FAB 버튼을 통해 상품 가격의 10%를 예약금으로 결제합니다. product_repository.dart의 reserveProduct 트랜잭션이 실행되어, 상품 상태가 reserved로 변경되고 구매자 ID(buyerId)가 기록됩니다.
현장 동일성 검증 (2단계): 구매자가 판매자를 만나 상품을 수령할 때, '현장 인수' 버튼을 누릅니다. ai_takeover_screen.dart가 활성화되고, 구매자는 현장 상품의 사진을 찍어 verifyProductOnSite 백엔드 함수를 호출합니다.
최종 거래 (3단계): AI가 원본 AI 보고서와 현장 사진의 동일성을 검증합니다.
○      검증 성공: 구매자가 '최종 인수 확정'을 누르면 completeTakeover가 호출되어 상태가 sold로 변경됩니다. (향후 PG사와 연동하여 잔금 90% 지급)
○      검증 실패: 구매자가 '거래 취소'를 누르면 cancelReservation가 호출되어 상태가 selling으로 롤백되고 buyerId가 제거됩니다. (향후 PG사와 연동하여 10% 예약금 환불)
4 신뢰 정책 및 유료화 기반 (개편안 1)  [1번은 논리 오류로 보류함.]
 
'끌어올리기' 어뷰징 방지: product_model.dart에 userUpdatedAt 필드를 도입했습니다. AI 검수/취소 등 시스템 변경은 updatedAt만 갱신하며, 오직 사용자가 직접 상품을 '수정/저장'할 때만 userUpdatedAt이 갱신됩니다. marketplace_screen.dart은 userUpdatedAt을 기준으로 정렬함으로써, 무료 '끌어올리기'를 원천 차단하고 향후 '유료 끌어올리기' 상품의 기반을 마련했습니다.
AI 검수 취소 및 재사용: 판매자는 AI 검수를 1회에 한해 취소할 수 있습니다 (aiCancelCount). 첫 검수는 isAiFreeTierUsed 플래그를 통해 무료로 제공됩니다. 만약 판매자가 취소 후, 상품의 설명이나 사진(aiReportSourceDescription, aiReportSourceImages)을 변경하지 않고 재검수를 요청하면, AI 비용을 절약하기 위해 유료(저가)로 기존 보고서를 재활성화(Reuse)합니다.
 
 
 


 

5. Nearby Shops / Toko Sekitar / 우리동네 가게
5.1 필요성 / Purpose
o   지역 소상공인을 위한 홍보 플랫폼으로, 사용자와 물리적으로 가까운 가게/서비스를 소개하는 메뉴입니다. Mozzy 커뮤니티와 지역 경제를 연결하며, 광고 수익화의 핵심 축을 담당합니다.
5.2 주요 구성 / Key Components
o   ●      가게 등록 구조:
o   ○      사업자 등록 여부와 관계없이 누구나 등록 가능.
o   ○      필수 정보: 가게명, 설명, 업종 카테고리, 대표 상품/서비스 (쉼표로 구분), 연락처, 운영시간, 다중 이미지 (최대 10장), 위치(GPS).
o   ●      업종 분류:
o   ○      사용자 검색 편의를 위해 명확한 업종 분류 제공.
o   ○      (예: 음식점, 카페, 마사지, 미용, 네일샵, 정비, 키즈, 병원, 기타 등)
o   ●      필터 및 정렬 기능:
o   ○      카테고리 필터: 업종 분류(예: '카페', '미용')에 따른 칩(Chip) 필터링.
o   ○      핵심 정렬:
o   ■           	기본순: 최신 등록순 (단, 스폰서 가게 최우선 노출).
o   ■           	거리순: 사용자 GPS 위치 기준 가장 가까운 순.
o   ■           	인기순: 조회수, 리뷰 수, 평균 평점을 조합한 '핫플' 순.
o   ●      노출 방식:
o   ○      사용자 위치 기반(행정구역 kab) 기본 필터링.
o   ○      카테고리별 리스트 + 정렬 옵션 제공.
o   ○      스폰서 가게: 유료 광고 상품 구매 시 리스트 최상단 고정 노출.
5.3 핵심 시스템 설명 / System Logic
o   ●      데이터 모델: 가게 정보(ShopModel)는 가게명, 설명, 업종(category), 대표 상품 리스트(products), 다중 이미지(imageUrls), 연락처, 영업시간, 위치(geoPoint), 인증 여부(trustLevelVerified) 등을 포함합니다.
o   ●      거리 및 위치 기반 정렬: 사용자 GPS(userModel.geoPoint)와 가게 GPS(shop.geoPoint)를 실시간으로 비교하여 '거리순' 정렬 및 카드에 거리(m/km)를 표시합니다.
o   ●      리뷰 및 평점 시스템 (핵심):
o   ○      사용자는 가게 상세 페이지에서 리뷰 작성 및 별점(1~5점) 제출이 가능합니다.
o   ○      ShopRepository는 트랜잭션(Transaction)을 통해, 리뷰 추가(addReview) 또는 삭제(deleteReview) 시 즉시 shops 컬렉션의 averageRating(평균 평점)과 reviewCount(리뷰 수)를 원자적으로 재계산하여 업데이트합니다.
o   ●      KPI 및 인기순: 상세 페이지(shop_detail_screen.dart) 진입 시, 해당 가게의 viewsCount(조회수)가 1 증가합니다. '인기순' 정렬은 이 조회수와 평점/리뷰 수를 복합적으로 계산합니다.
o   ●      신뢰도 표시: shop.trustLevelVerified가 true인 가게는 카드와 상세 페이지의 가게 이름 옆에 **'인증' 배지(Verified Badge)**가 노출됩니다.
o   ●      광고(스폰서) 로직: shop.isSponsored가 true인 가게는 모든 정렬 조건에 앞서 리스트 최상단에 노출되며, 카드에 '스폰서' 라벨이 부착됩니다.
o   ●      대표 상품/메뉴 표시: 등록 시 입력한 products 리스트는 상세 페이지 하단에 Chip 형태의 태그로 표시되어 '간이 메뉴판' 역할을 수행합니다.
o   ●      가게주 채팅 연동: 상세 페이지에서 '문의하기' 버튼 클릭 시, 구매자와 가게 주인(ownerId) 간의 1:1 채팅방(chat_service.dart)이 즉시 개설됩니다.

o   ●      DB 구조:
o   ○      /shops: 가게 정보 컬렉션.
o   ■           	(문서 필드 예: name, description, ownerId, locationName, geoPoint, locationParts, category, products (List<String>), imageUrls (List<String>), contactNumber, openHours, trustLevelVerified (bool), viewsCount (int), averageRating (double), reviewCount (int), isSponsored (bool), adExpiryDate (Timestamp))
o   ○      /shops/{shopId}/reviews: 가게별 리뷰 하위 컬렉션.
o   ■           	(문서 필드 예: userId, rating, comment, createdAt)
o   ●      기능 구현:
o   ○      Flutter(UI): local_stores_screen.dart (ListView, 필터, 정렬), shop_card.dart (요약 정보), shop_detail_screen.dart (상세 정보, 리뷰 UI), create/edit_shop_screen.dart (등록/수정 폼).

5.5 수익 모델 및 운영 계획 / Monetization & Ops
o   ●      기본 등록: 무료 (초기 사용자 확보).
o   ●      유료 광고 상품 (핵심 수익원):
o   ○      상단 고정 (Sponsor): isSponsored: true로 설정. 설정된 기간(adExpiryDate) 동안 리스트 최상단에 노출.
o   ○      '스폰서 가게' 라벨: 카드 UI에 광고 라벨 부착.
o   ○      (운영): 관리자 대시보드에서 isSponsored 및 adExpiryDate를 제어.
o   ●      사업자 인증 (Verified Badge):
o   ○      trustLevelVerified 필드를 활용. 사업자 등록증 등 서류 확인 후 '인증 배지' 부여 (유료 또는 무료 정책 선택 가능).
 
5.6 확장성 및 로드맵 / Scalability & Roadmap
o   ●      완료된 연동 기능:
o   ○      'Jobs' 연동: 가게 주인이 상세 페이지에서 '알바 구하기' 버튼 클릭 시, create_job_screen.dart으로 이동하며 가게 이름과 위치가 자동으로 입력됩니다.
o   ○      고객 응대: 가게 상세 페이지에서 고객과 1:1 채팅 기능이 구현되었습니다.
o   ●      향후 기능 (Roadmap):
o   ○      광고 성과 리포트: 소상공인(가게주)을 위한 클릭수(viewsCount), 채팅 전환율, 리뷰 통계 대시보드 제공.
o   ○      B2B 제휴: 로컬 배달 플랫폼 또는 POS 시스템과 연동.
o   ○      'POM' 연동: 사용자가 '뽐' 게시물 작성 시 가게를 태그(@)하고, 가게 상세 페이지에서 "이 가게를 태그한 뽐" 목록을 노출하여 소셜 증거(Social Proof) 강화.
o   [보류]○  	AI 활용: 사용자 리뷰 텍스트를 AI로 분석하여 키워드(예: "친절해요", "가성비")를 추출하고, 사용자 취향 기반 'AI 가게 추천' 기능 도입.
 

Nearby Shops 메뉴는 Mozzy의 지역 광고 수익의 핵심이며, 사용자에게 실질적 생활 정보도 제공합니다. 단순 디렉토리보다 진화된 ‘지역 가게 미니페이지 + 리뷰 + 프로모션’ 구조로 발전시킬 수 있습니다.


 

6. Groups / Grup Komunitas / 지역 모임  
1.1 필요성 / Purpose
o   지역 주민들이 취미, 운동, 종교 등 공통 관심사를 중심으로 함께 활동할 수 있는 커뮤니티 그룹을 제안·운영하는 메뉴입니다. 자발적인 커뮤니티 생성을 유도하여 사용자 체류 시간을 극대화하고 지역 기반 네트워킹을 활성화합니다.
1.2 주요 구성 / Key Components
o   ●      2-Tab UI 구성:
o   ○      모임 제안 (Proposals): "함께해요" 성격의 아이디어 제안 공간.
o   ○      활동 중인 모임 (Active Clubs): 정식으로 생성되어 활동하는 모임 목록.
o   ●      '모임 제안' 게시판 (ClubProposalModel):
o   ○      누구나 모임 아이디어 게시 가능 (예: 자카르타 조깅, 수공예 모임 등).
o   ○      핵심 필드: 모임명, 활동 내용, 위치, 카테고리, 대표 이미지.
o   ○      목표 인원 설정 (targetMemberCount): 제안자가 모임 생성에 필요한 최소 인원(예: 5명)을 설정합니다.
o   ○      참여 현황: 현재 참여 인원(currentMemberCount)과 참여자 목록(memberIds)이 실시간으로 표시됩니다.
o   ●      '활동 중인 모임' UI (ClubModel):
o   ○      하이브리드 UI:
o   1.         	(상단) 내가 가입한 모임: 사용자가 이미 가입한 모임 목록을 **수평 카로셀(Carousel)**로 제공하여 빠른 접근을 돕습니다.
o   2.         	(하단) 모든 모임 탐색: 광고(스폰서) 모임이 상단에 노출되는 **수직 리스트(List)**를 제공합니다.
o   ○      필터 및 정렬: 카테고리(운동, 예술 등) 필터, 거리순/인기순 정렬을 지원합니다.
1.3 핵심 시스템 설명 / System Logic
o   ●      모임 제안 및 참여 (/club_proposals):
o   ○      사용자는 '모임 제안하기'(create_club_screen.dart)를 통해 '목표 인원'을 설정하여 제안을 등록합니다.
o   ○      다른 사용자가 '제안 참여하기'(joinClubProposal)를 누르면 currentMemberCount가 1 증가합니다.
o   ●      그룹 자동 생성 (핵심 로직):
o   ○      joinClubProposal 실행 시, 현재 인원 >= 목표 인원이 되었는지 검사합니다.
o   ○      조건 충족 시, _convertProposalToClub 함수(Cloud Function 또는 Repository 트랜잭션)가 트리거됩니다.
o   ○      자동 전환 프로세스:
o   1.         	ClubProposalModel의 정보를 기반으로 새로운 ClubModel 객체를 생성합니다.
o   2.         	/clubs 컬렉션에 정식 모임을 등록합니다.
o   3.         	모임 전용 채팅방(chats/{clubId})을 자동 생성합니다.
o   4.         	기존 참여자(memberIds) 전원을 새 모임의 정식 멤버(/clubs/{clubId}/members)로 등록합니다.
o   5.         	원본 /club_proposals 문서를 삭제합니다.
o   ●      데이터베이스 구조:
o   ○      /club_proposals: 모임 제안 컬렉션.
o   ○      /clubs: 정식 모임 컬렉션.
o   ○      /clubs/{clubId}/members: 모임 멤버 하위 컬렉션.
o   ○      /clubs/{clubId}/posts: 모임 게시글 하위 컬렉션.
1.4 기술 아키텍처 / Technical Architecture

o   ○      /club_proposals: 제안 모임 (신규)

o   ○      getClubsStream(): '모든 모임 탐색' 탭 스트림 (스폰서 우선 정렬).

o   ○      DefaultTabController (또는 수동 TabController)를 사용하여 2개의 탭을 관리합니다.
o   ○      _buildProposalList: ClubProposalCard를 사용한 제안 목록.
o   ○      _buildActiveClubList: _buildMyClubsCarousel과 _buildAllActiveClubsList를 포함하는 하이브리드 UI.

7. Find Friends / Cari Teman Lokal / 동네 친구 (V2.2)

문서 버전: 2.2 (2025.12.09 개편 반영)

1. 개요 (Overview)
1.1 정의
관심사와 거주 지역(Kelurahan)을 기반으로 가까운 이웃과 연결되는 하이퍼로컬 소셜 매칭 & 커뮤니티 서비스.
1.2 개편 방향 (V2.2)
기존의 단순 프로필 나열 방식을 넘어, '목적 기반의 만남(동네 토크)' 기능을 도입하여 자연스러운 연결을 유도합니다.
●  	From: 사진 중심의 수동적 탐색 (데이팅 앱 유사)
●  	To: 주제(운동, 취미, 고민) 중심의 능동적 참여 및 그룹 활동(Multi-Chat) 지원.
●  	Trust: 프로필 완성도 검증 및 승인제 그룹 채팅으로 신뢰도 강화.

2. 핵심 기능 (Key Features)
2.1 탭 구조 및 탐색 (Dual Tab Structure)
사용자 편의를 위해 복잡했던 탭 구조를 **[탐색]**과 [내 친구] 2단 구조로 직관화했습니다.
Tab 1: 탐색 (Explore)
○  	상단 토글을 통해 두 가지 모드를 전환합니다.
○  	(A) 이웃 프로필 (Profiles): 기존 방식. 사진/관심사 기반으로 이웃을 둘러봅니다. (단, 필수 정보가 입력된 유저만 노출)
○  	(B) 동네 토크 (Dongne Talk): (New) "오늘 러닝하실 분?", "영어 회화 연습해요" 등 목적이 뚜렷한 게시글 피드.
Tab 2: 내 친구 (My Friends)
○  	내 친구: 서로 연결된 친구 목록.
○  	관심 이웃: 즐겨찾기(Star)한 이웃 목록.
2.2 동네 토크 & 멀티 채팅 (Post & Group Chat)
단순 1:1 대화뿐만 아니라, 주제를 가진 그룹 대화를 지원합니다.
●  	게시글 작성: 태그(Tag), 내용, 최대 참여 인원, 채팅 유형(1:1 vs 그룹)을 설정하여 작성.
●  	채팅 유형:
○  	1:1 대화: 신청 즉시 대화가 시작되거나 승인 후 연결.
○  	그룹 채팅 (Multi-Chat): 최대 10명까지 참여 가능. **'승인제'**로 운영되어 방장(작성자)이 멤버를 관리함.
2.3 승인 및 멤버 관리 시스템 (Approval System)
무분별한 채팅 참여를 방지하고 안전한 모임을 보장합니다.
●  	참여 요청: 사용자가 그룹에 [참여 요청]을 보냄 → 상태가 waiting으로 변경.
●  	승인/거절: 방장은 [참여 요청 관리] 메뉴에서 신청자의 프로필(사진, 소개, 신뢰도)을 확인 후 승인 또는 거절.
●  	자동 초대: 승인 시 해당 유저는 자동으로 그룹 채팅방에 초대되며, "OOO님이 참여했습니다" 시스템 메시지가 전송됨.
●  	종료 처리: 게시글 삭제 시 연결된 채팅방의 참여자가 초기화되어 대화가 종료됨(Soft Close).
2.4 강화된 프로필 및 신뢰 정책 (Enhanced Trust Profile)
'유령 유저'와 '불량 유저'를 필터링하기 위해 프로필 정책을 강화했습니다.
●  	노출 조건 (Quality Control):
○  	다음 조건이 충족되지 않으면 '친구 찾기' 리스트에 노출되지 않음 (isProfileReadyForMatching).
○  	조건: 프로필 사진, 닉네임, 자기소개(5자 이상), 관심사(1개 이상) 필수.
●  	통합 프로필 설정:
○  	기본 정보(필수) + 신뢰 인증(전화번호) + 소셜 프로필(선택)이 하나의 화면(ProfileSetupScreen)에서 관리됨.
●  	프라이버시 보호:
○  	상세 주소(Street/RT/RW)는 절대 노출되지 않으며, 행정구역(Kel/Kec) 단위까지만 표시 (AddressFormatter 적용).

3. UI/UX 흐름 (User Flow)
3.1 게시글 작성 (Create Post)
[탐색] 탭 → [동네 토크] 모드 선택.
FAB(글쓰기 버튼) 클릭.
내용 입력, 태그 추가, '그룹 채팅' 활성화 여부 및 '최대 인원' 설정.
작성 완료 시 피드에 카드 형태로 노출.
3.2 그룹 참여 (Join Group)
피드에서 관심 있는 '동네 토크' 카드 발견.
[그룹 참여 요청] 버튼 클릭 (이미 참여 중이면 [대화 입장] 버튼 노출).
요청 상태(waiting)가 되면 버튼이 **[승인 대기 중 (취소)]**로 변경됨.
방장이 승인하면 [대화 입장] 버튼이 활성화되고 채팅방 접근 가능.
3.3 멤버 관리 (Manage Members - 방장)
내 게시글 카드에 [승인 대기 N명] 뱃지 버튼 표시.
버튼 클릭 시 [참여 요청 관리] 바텀 시트 오픈.
신청자 목록에서 프로필 사진을 눌러 상세 정보 확인.
(V) 승인 또는 (X) 거절 버튼 클릭 → 결과에 따라 즉시 채팅방 초대 및 UI 갱신.

“뽐” 메뉴는 단순한 콘텐츠 게시판이 아니라, 사용자의 자존감과 지역 애착을 연결하는 감정 중심의 구조입니다. Mozzy이 제공하는 ’따뜻한 연결감’을 가장 잘 구현하는 메뉴가 될 것입니다.
●      이하 구현 기획은 “POM 브랜드 컨셉 가이드 (뽐의 정체성 설계).docx  참조”

11. 함께 해요 / Together / Ayo Bareng
1. 개요 (Overview)
●      정의: 동네 이웃과 가볍고 목적 지향적인 만남을 주선하는 '디지털 전단지(Digital Flyer)' 기반의 번개 모임 서비스.
●      기획 의도: "친구는 부담스럽고, 혼자는 심심한" 현대인의 심리를 공략. 거창한 동호회(Clubs)나 1:1 친구 찾기(Find Friends)와 차별화된,'행위(Action) 공유' 중심의 느슨한 연결을 지향함. (예: "같이 걷기", "잠깐 담배 타임")
●      슬로건: "지금, 같이 할래요?" (Let's do it together)
2. 핵심 기능 (Key Features)
2.1 디지털 전단지 (Digital Flyer UI)
●      디자인: 형광색(Neon), 종이 질감(Paper), 다크 모드(Dark) 등 눈에 띄는 전단지 테마를 적용하여, 길거리 벽보를 보는 듯한 아날로그 감성을 디지털로 구현.
●      정보 전달: 복잡한 설명 대신 What(무엇을), When(언제), Where(어디서) 3가지를 굵고 명확하게 전달.
2.2 지도 기반 위치 설정 (Pin-point Location)
●      작성자: "현재 위치"를 강제하지 않고, 구글 맵(Google Map)을 직접 움직여 **실제 모임 장소에 핀(Pin)**을 꽂는 직관적인 방식 제공.
●      참여자: 상세 화면에서 미니맵(Mini Map)을 통해 정확한 모임 위치를 시각적으로 확인 가능.
2.3 QR 티켓 시스템 (Together Pass)
●      참여(Join): '참여하기' 버튼 클릭 시, 서버 트랜잭션을 통해 즉시 **'QR 입장권'**이 발급되어 [마이 Mozzy > 티켓] 탭에 저장됨.
●      인증(Verify): 모임 현장에서 주최자(Host)가 참여자의 QR 코드를 앱 내 **스캐너(Scanner)**로 인식하여 상호 신원 확인 및 출석 체크 수행.

3. 사용자 흐름 (User Flow)
탐색 (Discover):
○      홈 화면: 최상단 카로셀(First Carousel)에서 전단지 카드를 가로로 넘겨보며 탐색. (아이콘 1.2배 강조됨)
○      전용 탭: '함께 해요' 메뉴 진입 시 전체 목록 확인.
생성 (Create):
○      하단 중앙 [+] 버튼 클릭 → 디자인 테마 선택 → 제목/내용 입력 → 지도에서 위치 핀 설정 → 시간 및 인원(슬라이더) 설정 → 생성 완료.
참여 (Join):
○      상세 화면에서 [함께 하기] 버튼 클릭 → 참여 완료 팝업 및 QR 티켓 자동 발급.
확인 (Check):
○      참여자: [마이 Mozzy > 티켓]에서 QR 코드 제시.
○      주최자: 상세 화면 하단 [QR 스캔하기] 버튼으로 카메라 구동 → 참여자 QR 스캔 → "OOO님 환영합니다" 메시지 확인.

4. 기술 사양 (Technical Specs)
●      데이터 모델 (TogetherPostModel):
○      geoPoint: 지도 핀 좌표 (Firestore GeoPoint)
○      qrCodeString: 모임 고유 식별 토큰 (UUID)
○      designTheme: 전단지 스타일 키 (neon, paper, dark)
○      maxParticipants: 최대 인원 (2~10명, 슬라이더 제어)
○      imageUrl: 업로드된 이미지 URL (Firebase Storage)
●      보안 및 검증:
○      QR 데이터 포맷: JSON {'uid': userId, 'pid': postId, 'token': qrCodeString}
○      스캔 로직: QR 내 pid와 현재 모임 ID 일치 여부 검증 후 사용자 정보 로드.
 

 
 

II.기술 아키텍처 (Technical Architecture)
1.1 전체 구조 개요 (Overview)
Mozzy(모지)는 Flutter 기반의 크로스 플랫폼 앱으로, 백엔드는 Firebase 중심의 서버리스 구조를 사용합니다. AI 검수 기능은 외부 이미지 분석 API와 커스텀 모델을 통해 연동되며, 실시간 채팅, 알림, 위치 기반 서비스까지 통합 구현된 통합형 슈퍼앱 구조입니다.

  └─ 거래 확정 → 자동 정산 / 환불

1.4 기능별 연계 흐름 (Key Flows)
●       AI 검수 등록 플로우
o   사용자가 이미지와 필수정보 입력 → AI -API 호출 → 검수 결과 & 시세 제공 → 동의 후 게시
●       거래 에스크로 플로우
o   구매자 예약금 결제 → Firebase 트리거로 보관 상태 → 인수 확인 후 잔금 결제 + 자동 정산
●       그룹 자동 생성 플로우
o   제안글 등록 → 가입자 수 체크 (예: 10명) → Cloud Function이 그룹 게시판 자동 생성
●       지역 기반 노출 알고리즘
o   사용자 위치 → 각 메뉴별 게시물 거리 정렬 + 지역 맞춤 노출 최적화

1.5 보안 및 신뢰 시스템 (Security & Trust Layer)
●       Firebase Authentication: 전화번호 + 이메일 기반 다중 인증
●       사용자 신뢰지수 시스템: 후기 + 신고 이력 기반
●       자동 만료/필터링: Cloud Scheduler 기반 게시물 관리
●       신고처리 → 관리자 개입 + 자동 제한 시스템 (게시 제한, 노출 제한 등)

Mozzy(모지)의 기술 구조는 개발 인력 최소화, 운영 효율화, 유연한 기능 확장을 모두 고려한 Firebase 중심 구조입니다. AI 기능과 결제, 지도, 채팅 등 모든 실생활형 서비스가 하나의 앱에서 동작하도록 설계되었습니다.
